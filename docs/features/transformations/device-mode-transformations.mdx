When you <Link to="#adding-a-transformation">add a transformation</Link> and connect it to a destination supporting the <Link to="/destinations/rudderstack-connection-modes/#device-mode">device mode</Link>, the following steps occur in the backend:

1. RudderStack SDK sends the events to the routing server.
2. Routing server sends those events to the transformation server.
3. Transformation server applies transformation to the events and returns the transformed events to the routing server.
4. Routing server returns the transformed events to the SDK.
5. SDK sends the transformed events to the specified device mode destinations.

## Limitations

The following limitations are applicable when invoking a transformation in the device mode:

- The network availability and server response can lead to latency while sending the events. Hence, there might be some delays before the events are reflected in the destination SDKs.
- The iOS SDK does not support the background processing of an event when the app is closed. However, the backlog event is sent the next time the app is opened.
- The following memory and time limits must be followed:

    | Parameter    | Limit     |
    | :----------- | :-------- |
    | Memory limit | 128 MB      |
    | Execution time limit   | 4 seconds |
