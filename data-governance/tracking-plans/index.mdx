# Tracking Plan API

Tracking Plans are RudderStack's latest offering within the [**Data Governance**](https://rudderstack.com/docs/data-governance/) feature that allow you to plan and decide what your event data should look like - **before** it starts flowing through. This essentially means that upon capture, your event data will always conform to a fixed, predefined schema.

## Tracking Plan features

Tracking Plans allow you to:

* Define specific event names and properties for each of your `track`, `group`, and `identify` calls.
* Specify whether a property/attribute is required and assign a particular data type to it.
* Have better control of your data streaming by supporting a versioning system.
* Leverage the [**Data Governance API**](https://rudderstack.com/docs/data-governance/rudderstack-data-governance-api/) to evaluate your inbound events, payloads, and metadata and compare them with your plans.

<div class="successBlock">

You can also use the <a href="https://rudderstack.com/docs/data-governance/rudder-typer/">RudderTyper</a> tool alongside the Tracking Plans. <strong>RudderTyper</strong> is a tool for generating strongly-typed RudderStack analytics library wrappers based on your published Tracking Plan specs.
</div>

<div class="infoBlock">

Read this <a href="https://rudderstack.com/blog/dogfooding-at-rudderstack-tracking-plans-part-1/">RudderStack blog</a> to know more about the Tracking Plans feature and how to use it effectively.
</div>

## Authentication

The Tracking Plan API uses **Basic Authentication** for authenticating all the requests.

<div class="successBlock">

All the popular HTTP clients (e.g., CURL, Postman, HTTPie) have default support for Basic Authentication.
</div>

The Basic Authentication for this API requires a username and password, where:

* The username is the **email address** you used to sign up for RudderStack.
* The password is the **personal access token**.

<div class="infoBlock">

For more information on obtaining the personal access token, refer to <a href="https://rudderstack.com/docs/transformations/api-access-token/">these docs</a>.
</div>

For example, if the email address is `name@surname.com` and the personal access token is `1zl4GJkLu0HsBdrDK88NgYZzY2E`, your request must have the following HTTP header:

```curl
Authorization: Basic bmFtZUBzdXJuYW1lLmNvbToxemw0R0prTHUwSHNCZHJESzg4TmdZWnpZMkU=
```

<div class="infoBlock">

  In case of the <strong>Invalid Authorization Header</strong> error, verify if the email address and the personal access token are valid.
</div>

<div class="successBlock">

  You can use this <a href="https://www.blitter.se/utils/basic-authentication-header-generator/">Basic Authentication Header Generator</a> to generate the HTTP header.
</div>

## Tracking Plan model

| **Attribute** | **Type** | **Requirement** | **Description** |
|:--------------|:----------|:-------------------|:-----------------|
|`name` | ID | Read-only | Auto-generated name |
| `display_name` | String | Required | Display name of the tracking plan. | 
| `rules` | Rules | Optional | Schema rules for the tracking plan. |
| `version` | Version | Read-only | Current version of the tracking plan. | 
| `create_time` | Datetime| Read-only | Refers to the time when the tracking plan was created. |
| `update_time` | Datetime | Read-only | Refers to the time the tracking plan was updated. |


## Rules Model

| **Attribute** | **Type** | **Requirement** | **Description** |
|:--------------|:----------|:-------------------|:-----------------|
| `events` | Array | Optional | Rules applied to the `track` calls. |
| `identify` | Object | Optional | Rules applied to the `identify` calls. |
| `group` | Object | Optional | Rules applied to the `group` calls. |


## Events Model

| **Attribute** | **Type** | **Requirement** | **Description** |
|:--------------|:----------|:-------------------|:-----------------|
| `name` | String | Required | Event Name. |
| `description` | String | Optional | Refers to the event description. |
| `rules` | JSON | Required | Rules applied to the event.
