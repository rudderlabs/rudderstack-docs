# Data Regulation API

RudderStack's Data Regulation API allows you to delete the collected data, suspend data collection, or do both - for a particular user identified by a `userId`.

This guide covers the data regulation feature in detail as well as details the various endpoints of the Data Regulation API.

[![Plan: Enterprise](https://img.shields.io/static/v1?label=PLAN&message=ENTERPRISE&color=blueviolet&style=for-the-badge)](https://rudderstack.com/enterprise-quote)

## What is data regulation?

At RudderStack, we respect the users' privacy choices and are committed to keeping their data secure at all times.

You can leverage RudderStack's data regulation feature to suppress any incoming data for a given user. RudderStack will drop the events for that user - these events will not be shown in any of the debuggers and will not be forwarded to any destinations. You can also use the feature to direct RudderStack to delete any collected data in a given destination.

<div class="warningBlock">

Once a user is suppressed, replaying the user events is not possible.
</div>

## Prerequisites

To use the Data Regulation API successfully, you need to generate a [**Personal Access Token**](https://rudderstack.com/docs/transformations/api-access-token/). This is required to authenticate the API.

## API authorization

The Data Regulation API uses the **Bearer Token Authentication** for authenticating all the requests.

<div class="infoBlock">

The Personal Access Token should be used as the bearer token for authentication.
</div>

## Adding a new data regulation

This request allows you to add a new data regulation required to suppress/delete/unsuppress a given user's data.

* **Request Type**: **POST**

* **Request Format**: 

```bash
https://api.rudderstack.com/v2/regulations
```

* **Request Body**:

```json
{
  "regulationType": "suppress/suppress_with_delete",
  "destinationIds": [],
  "sourceIds": [],
  "users": [{
    "userId": "<user_ID>",
    "phone": "<phone_number>",
    "email": "<user_email>"
  }]
}
```
### `regulationType`

This field lets you define the type of data regulation you want to add. It supports the following values:

| Parameter Name | Description |
| :-----------------| :-----------|
| `suppress`           | Directs RudderStack to suppress the incoming user data. |
| `suppress_with_delete` | Directs RudderStack to suppress any incoming user data and delete any events from the destinations configured by the user. |

### `destinationIds` and `sourceIds`

You can set these **optional** fields to specify the sources/destinations from which the data should be suppressed or deleted. 

<div class="warningBlock">

<ul>
<li>Set the <code class="inline-code">sourceIds</code> field when setting the <code class="inline-code">regulationType</code> to <code class="inline-code">suppress</code>.</li>
<li>Set the <code class="inline-code">destinationIds</code> field when setting the <code class="inline-code">regulationType</code> to <code class="inline-code">suppress_with_delete</code>.</li>
</ul>

<strong>Avoid specifying both <code class="inline-code">sourceIds</code> and <code class="inline-code">destinationIds</code>.
</div>

### `users`

The API lets you specify the users for whom you want to set the data regulation. These users can be identified either by their `userId`, `phone`, or `email`.

<div class="infoBlock">

<strong>Only <code class="inline-code">userId</code> is a mandatory field. The other fields are optional as some destinations require fields like phone number or email to associate a given user and do not strictly associate a user by their user ID.
</div>

<div class="warningBlock">

We recommend that you provide the user's <code class="inline-code">phone</code> and <code class="inline-code">email</code> <strong>at your own discretion and only if it is required</strong>.
</div>

* **Sample Request**

```bash
curl --location --request POST 'https://api.rudderstack.com/v2/regulations' \
--header 'Authorization: Bearer 22aJ7iEeWud78wDaR6dbAe29Nw2' \
--header 'Content-Type: application/json' \
--data-raw '{
    "regulationType": "suppress",
    "sourceIds": [
        "src-1"
    ],
    "users": [
        {
            "userId": "123",
            "phone": "9876543210",
            "email": "name@surname.com"
        }
    ]
}'
```

* **Expected Response**

```text
Created
```

### List all data regulations

This request lists all your data regulations.

* **Request Type**: **GET**

* **Request Format**: 

```bash
https://api.rudderstack.com/v2/regulations
```

* **Parameters**:

| Parameter         | Type | Description |
| :---------------| :------------| :------------------|
| `after_cursor` | String | Should be used with the data after the given cursor position. This can be obtained from the `paging` object. Refer to the sample request below for more information. |
| `per_page` | Integer | The number of data regulation items to be returned. |

* **Sample Request**

```bash
curl --location --request GET 'https://api.rudderstack.com/v2/regulations' \
--header 'Authorization: Bearer 22YI2pCURNbcG0fGRfkgAdcWQsW'
```

## Cancel a data regulation

This request cancels an existing data regulation.

* **Request Type**: **DELETE**

* **Request Format**: 

```bash
https://api.rudderstack.com/v2/regulations/{regulation_id}
```

* **Parameters**:

| Parameter         | Type | Description |
| :---------------| :------------| :------------------|
| `regulation_id` | String |  The ID of the data regulation to be cancelled. |

* **Sample Request**:

```bash
curl --location --request GET 'https://api.rudderstack.com/v2/regulations' \
--header 'Authorization: Bearer 22YI2pCURNbcG0fGRfkgAdcWQsW'
```
